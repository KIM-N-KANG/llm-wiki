## 한줄 요약

AI가 혼자서도 잘 일할 수 있는 작업 환경(Harness)을 구조·맥락·계획·실행·검증·개선의 6개 축으로 설계하는 방법론.

## 핵심 내용

- **Harness Engineering 정의**: 프롬프트 엔지니어링(Level 1) → Context Engineering(Level 2) → Harness Engineering(Level 3)으로 진화. AI에게 말을 잘 하는 것이 아니라 "일하는 방법을 설계해주는 것"
- **6개 순환 축**: 구조(Scaffolding) → 맥락(Context) → 계획(Planning) → 실행(Orchestration) → 검증(Verification) → 개선(Compounding) → 다시 구조로
- **구조(Scaffolding)**: Monorepo 관리, 역할별 폴더링(src/docs/tests/.claude/), 사람 문서(docs/)와 AI 기록(.dev/) 분리, Skills·Hooks·Agents·MCP·Plugins 도구 배치
- **맥락(Context Engineering)**: CLAUDE.md 3단계 상속(User→Project→Folder), Progressive Disclosure로 필요한 것만 필요할 때, .claude/rules/ glob 패턴 조건부 로드
- **계획(Planning)**: "해줘" 대신 "같이 계획 세우자" → AI가 계획 작성 → 사람이 승인 → 실행. AskUserQuestion으로 모호함 제거. 커스텀 Plan 스킬(/specify, /deep-interview, /clarify)
- **실행(Orchestration)**: 단일 에이전트(90% 상황), 병렬 Subagent(독립 작업), Team Mode(~7x 토큰 비용). Ralph Loop(완료 기준 합의 후 반복), autoresearch(밤새 자율 실험)
- **검증(Verification)**: Sprint Contract(작업 전 완료 기준 합의), Generator-Evaluator 분리(같은 컨텍스트에서 만들고 평가하면 안 됨), 안전장치(git worktree, Runtime Gate, --dry-run)
- **개선(Compounding)**: 3번 반복 작업 → Skill로 자동화, 3번 같은 실수 → Rule 추가, 안 쓰는 것은 즉시 삭제

## 주목할 인사이트

- "모델 교체로 5% 개선하는 것보다 하네스 설계로 15% 개선하는 것이 현실적이다"
- Anthropic 인용: "규율은 코드가 아니라 스캐폴딩에서 드러난다"
- "자신의 시스템에 대한 이해 없이 만든 Harness는 AI Slop 생산 기계가 된다" — 먼저 자기를 알라
- Harness의 핵심: 맥락(뭘 해야 하는지) + 제한(뭘 하면 안 되는지) + 작업 흐름(어떻게 일하는지) + 검증(잘 했는지 확인)
- Generator와 Evaluator는 반드시 분리해야 함 — 같은 컨텍스트에서 만들고 평가하면 quality가 mediocre해도 자신있게 칭찬함
- "Harness의 공간은 모델이 좋아져도 줄어들지 않는다. 이동할 뿐이다." (Anthropic)
